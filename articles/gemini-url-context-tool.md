## TL;DR（まず結論）

**URL Context Tool は“URLを渡すだけでモデルが本文を読み、根拠として参照してから回答する”仕組み**です。ファイル貼り付けやスクレイピングの前処理が要らず、**導入が簡単**、かつ**トークン消費を大幅に抑えられる可能性が高い**のが強みです。PDFや長文ページ、複数URLの比較で効果が出やすく、検索のグラウンディングと併用すれば「広く探す＋深く読む」をひと呼び出しで実現できます。

---

## URL Context Tool とは？（一般向けのやさしい説明）

- プロンプトに URL を書くだけで、そのページの**本文（HTML/JSON/CSV/XML、PDF、画像など）をモデルが読んだ上で**回答します。
- 1 回で**最大20 URL**まで指定でき、**1 URL あたり最大約34MB**まで扱えます。
- 参照したURLや取得ステータスは**メタデータ**として返り、どこを根拠にしたか確認できます。
- Google 検索グラウンディングと併用すると、最新トピックも「検索→本文読解」の流れで対応できます。

> 要するに「**URLを根拠に、本文を実際に読んでから答える**」ための標準機能です。

---

## なぜ“有用”なのか（冒頭の要点）

0) **トークン消費を下げやすい**：全文を貼る代わりに URL を参照させるため、**プロンプトのトークン量を大幅に抑えられる可能性が高い**。特にPDF／長文記事／複数ページ比較で効く。

1) **導入が簡単**：URL を渡すだけ。ファイルアップロードやスクレイピング設定が不要。

2) **実務で効く精度**：本文（PDFや画像含む）まで読みに行くので、要約や比較の精度を上げやすい。根拠URLもメタデータで確認でき、説明責任を果たしやすい。

3) **スケールしやすい**：複数URLを一度に読み合わせられるため、リサーチやドキュメント比較の自動化に向く。

4) **検索と組み合わせて“広く深く”**：検索で網羅性を確保し、URL Context で本文理解の深さを担保するワークフローがシンプル。

---

## どんなシーンで便利？（使用例）

- **プロダクト比較・競合調査**：製品ページ、ヘルプ、リリースノートを複数URLで読み合わせ、差分と要点を短時間で整理。
- **レポート要約・サマリ生成**：PDFの白書や調査資料をURL指定で要約し、箇条書きや図表の説明を抽出。
- **技術ドキュメントQ&A**：APIドキュメント／README／Issue をURLで渡し、設定手順やサンプルの穴埋めを生成。
- **価格や仕様の抽出**：料金表ページから価格・プラン・制限を抽出して一覧化。
- **社外サイトを根拠としたライティング**：引用元をURLで固定し、ブログ初稿やプレス向け要約を生成。

> さらに具体的な「使い方テンプレ」は、この章のあとに続く **「追加のユースケース（テンプレ集）」** に20パターンまとめてあります。

## 使い方のイメージ（一般向け）

- プロンプトに「このURLとこのURLを比較して、差分を表にして」と書き、URL Context を有効化。
- 併せて Google 検索のグラウンディングをオンにすると、最新トピックでも“検索→本文読解”の二段構えで対応。
- 結果には、参照したURLと取得ステータスがメタデータとして含まれます（検証しやすい）。

**メモ**：URL Contextは**プロンプト総トークンに加算**されます。**とはいえ、全文貼付方式よりはトークンを大幅に減らせる可能性が高い**ため、まずはURL参照で試す価値があります。長大なページやURL数が多い場合は、必要URLに絞る／分割呼び出し／要点抽出の併用を検討しましょう。

## Google AI Studio（Playground）での設定（一般向け・簡潔）

1. **新しいプロンプトを作成**（Playground）。
2. 右側の**設定パネル**で **Context / Tools** 相当の項目を開き、**URL Context（またはURLの追加）をオン**。
3. **URLを追加**（最大20件）。PDF/画像も可。**認証が必要なページは不可**（後述）。
4. 必要に応じて **Google 検索のグラウンディング**をオン。
5. プロンプトに**やりたいことと評価軸**（例：比較表の列、優先順位、根拠表示）を書き、実行。

> UIラベルは環境で多少異なる場合があります。概ね「右パネル → コンテキスト／ツール → URL を追加」で探せます。

## 注意点・制限（一般向け）

- **URL数**：1 回の呼び出しで最大 20。
- **サイズ**：1 URL の取得上限は約 34MB。
- **認証が必要なページは非対応**：ログインが必要な会員ページ、Cookie/セッション前提の画面、Basic認証、OAuth/トークンが必要なAPIエンドポイント等からは**直接取得できません**。公開アクセス可能なURLが前提です。必要に応じて、FireCrawlや自前フェッチで取得→要点抽出してからモデルに渡す構成を検討。
- **非対応の種類**：Paywallの裏側、YouTube動画、Google Workspace（Docs/Sheets等）の共有リンク、音声・動画ファイルは直接対象外。
- **新鮮さ**：環境（API/Vertex AI）によっては**ライブ取得ではなくキャッシュ優先**の挙動があります。速報性が要る場合は、検索グラウンディング併用や別経路での取得も検討を。

# ここから先は“より深く知りたい人向け”（技術ブロガー視点）

## URL Context Tool と Web Fetch / FireCrawl MCP の違い（体系的に比較）

### アーキテクチャ思想

- **URL Context Tool**：モデル呼び出しの一部として、URLの取得〜解析までを**Gemini側が内製**。メタデータで「どのURLを読んだか」を返す。
- **Web Fetch / FireCrawl MCP**：**取得レイヤーを自前運用**。スクレイピングのオプション（JSレンダリング、`waitFor`、`include/exclude`、UA偽装、並列・バッチ等）を細かく制御し、結果（Markdown/JSONなど）を**明示的にモデルへ渡す**。

### 機能差

- **ダイナミックレンダリング**：FireCrawl MCP は JS レンダリングや待機が可能。URL Context の挙動はシンプルで細かなレンダリング制御は想定外。
- **バッチ／サイト横断**：FireCrawl MCP は `batch_scrape`、`crawl`、`map` 等の**収集向け機能**が充実。URL Context は「与えたURLを読む」ことに特化。
- **検証メタデータ**：URL Context は `url_context_metadata` を返す。FireCrawl MCP もログ/ステータスでトレース性を確保可能（設計次第）。

### コスト設計

- **URL Context**：取得〜読解が一体化して簡潔。本文は**入力トークンに加算**されるため、長大ページ／多URLでは分割や要点抽出で最適化。
- **FireCrawl MCP**：事前に**抽出・整形**（必要箇所だけMarkdown/JSON）してから投入できるため、**無駄トークンを抑制**しやすい。

### 品質・再現性

- **URL Context**：公式の標準機能として安定。検索グラウンディングと併用で「広く→深く」の再現性高いパターンが作りやすい。
- **FireCrawl MCP**：取得パイプラインが**コードとして残る**（待機・再試行・並列など）。**再現性・可搬性**の高い“収集レシピ”を作る用途に向く。

### 早見表（要点）

| 観点 | URL Context Tool | Web Fetch / FireCrawl MCP |
|---|---|---|
| セットアップ | URL を渡すだけ（モデル側で取得） | MCPサーバーを接続し取得設定を運用 |
| 制御粒度 | 低〜中（標準化重視） | 高（レンダリング/並列/タグ制御/バッチ） |
| 典型ユース | 複数URLの読解・比較・要約 | 大規模収集・動的サイト・精密抽出 |
| スケール | 最大20URL／約34MB/URL | 任意（自前のクロール/バッチで拡張） |
| 検証 | `url_context_metadata` | ログ・ステータスを自前で管理 |
| コスト | 本文が入力トークンに加算 | 先に抽出整形→必要最小限だけ投入 |

### どちらを選ぶべき？

- **まず最短で動かしたい／下準備を減らしたい** → **URL Context Tool**
- **収集含めて作り込みたい／JS描画やバッチ運用が必須** → **FireCrawl MCP（または同種のWeb Fetch）**
- **速報性と根拠の両立** → **Google検索グラウンディング＋URL Context**

### 併用パターン（実務向け）

- **FireCrawl→要点抽出→URL Context**：先に構造化抽出して短く要約→URL Contextで“精読＋最終生成”。
- **検索→候補URL→URL Context**：検索で候補選定→本文読解で精度を上げる。
