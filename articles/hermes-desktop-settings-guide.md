## TL;DR

- Nous Research の [**Hermes Desktop**](https://hermes-agent.nousresearch.com/desktop) の設定 14 カテゴリ・全 186 項目を、実機監査とソースコード照合で検証した非公式日本語ガイドを公開した → **https://hermes.ai-deck.app**
- 検証は v0.16.0 / コミット `d165933` に固定。Computer Use で設定 14 画面のアクセシビリティツリーを採取し、既定値は設定 API とソースで裏取りした
- AI エージェントの設定は安全性に直結する。承認モード・シークレットマスキング・プライベート URL あたりは、一度全項目に目を通しておくことを勧める

## なぜ作ったか — 紹介記事はあるのに、リファレンスがない

いま、[Hermes Agent](https://github.com/NousResearch/hermes-agent) がすごい勢いで広がっています。2026年2月25日の公開から 4ヶ月足らずで GitHub スターは約 19 万（2026年6月10日時点）。6月初旬には公式デスクトップアプリ **Hermes Desktop** が macOS / Windows / Linux 向けに公開され、ターミナルを使わない層にも一気に届くようになりました。

日本語の情報も増えています。ただし内訳を見ると、「インストールして触ってみた」系の紹介記事がほとんど。その次の段階——設定画面を開いて「承認モードの Smart と Manual は何が違う？」「ゲートウェイって何？」と手が止まった人の受け皿が見当たりませんでした。

公式ドキュメントは英語と簡体字中国語のみで、日本語版はありません。しかも開発スピードが速く、ドキュメントが画面の全項目に追いついていません。つまり「公式を読めば分かる」が成立しないんです。

普通のアプリなら、意味の分からないトグルがあっても大して困りません。でも AI エージェントの設定は事情が違います。シェルコマンドの自動承認、内部ネットワークへのアクセス、シークレットの扱い——安全性に直結する項目が当たり前のように並んでいます。意味を知らないまま既定値を変えるのは、フォームを知らずに高重量を扱うようなもの。しばらくは動きますが、どこかで腰をやります。

それなら、設定だけは全項目を根拠付きで引ける場所を作ろう。そう決めて、デスクトップアプリ公開から 1 週間以内のビルドを対象に検証を始めました。

## 作ったもの

**Hermes Desktop ガイド（非公式）**: https://hermes.ai-deck.app

- 設定 14 カテゴリ（モデル / チャット / 外観 / ワークスペース / 安全性 / メモリとコンテキスト / 音声 / 詳細 / プロバイダー / ゲートウェイ / ツールとキー / MCP / アーカイブ済みチャット / 情報）をすべてページ化
- 各項目に **画面表示・内部キー・製品既定値・推奨値** を記載
- [全設定項目索引](https://hermes.ai-deck.app/reference/settings-index/) は 16 表・186 行
- はじめに / 基本操作 / トラブルシューティングのセクションもあるが、核は設定リファレンス

対象は、初めて Hermes Desktop を入れる人から、全項目の既定値と根拠を確認したい人まで。

## どう検証したか — バージョン固定・実機監査・ソース照合

既定値や推奨値を載せる以上、「どのバージョンで、何を根拠に」が言えないと意味がありません。3 つのルールで進めました。

### 1. バージョンを固定する

対象は v0.16.0、コミット `d165933`、実機確認日は 2026-06-07。確認の時点でアプリにはさらに新しい更新の通知が来ていましたが、未確認の更新内容は混ぜず、実際に起動して確認したコミットに固定しました。根拠の詳細は[バージョンと調査根拠](https://hermes.ai-deck.app/reference/version-evidence/)のページにまとめています。

### 2. 実機を Computer Use で監査する

設定サイドバーを上から順に、14画面とサブメニュー 4つを開いていきます。採取するのは画面表示とアクセシビリティツリーだけ。トグルも入力欄も保存ボタンも操作しません。読むだけの監査です。

実機を見ないと分からないことは、実際にありました。たとえば「ツールとキー」画面は、内部キー末尾の `_API_KEY` や `_TOKEN` を省略して表示します。`GATEWAY_PROXY_KEY` は画面上では **GATEWAY PROXY**。ソースだけ読んでも画面の表記と突き合わせられませんし、画面だけ見ても内部キーが分かりません。両方を 36 行分対照して、ようやく表になりました。

### 3. 既定値はソースと API で裏取りする

画面に表示されているのは「現在の値」であって「既定値」ではありません。既定値は、設定 API の `DEFAULT_CONFIG` と `/api/config/defaults` の返却経路、外観系は `localStorage` 読み込み時のフォールバック、それに同一コミットのソースコードと日本語翻訳ファイルを突き合わせて確定させました。実画面と一般ドキュメントが矛盾した場合は、対象ビルドの実画面とソースを優先しています。

## 設定を読むと、設計思想が見えてくる

186項目を順に潰していくと、このアプリが「自律性と安全性のトレードオフをユーザーに委ねる」設計だと分かってきます。象徴的な項目を 3つだけ挙げます。

**承認モードは manual / smart / off の 3 段階**。既定は `manual` で、エージェントが実行するコマンドを人が確認します。`smart` は補助モデルがリスクを判定して自動承認を支援します。`off` は承認プロンプト自体を省略します——ただし致命的な破壊操作のハードブロックなど、一部の防御は残ります。残るとはいえ人の確認を失うので、`off` は「安全」と同義ではありません。

**ファイルチェックポイント（既定オフ）**。オンにすると、ファイル変更の前に、実プロジェクトの `.git` とは別のシャドウ Git ストアへスナップショットを作ります。エージェントにコードを触らせるなら、オンをおすすめします。

**シークレットを伏せる（既定オン）**。API キーやトークンに見える文字列を、ツール出力が会話コンテキストやログへ入る前にマスキングします。ただし完全な漏えい防止を保証する機能ではないので、これに頼り切らず、平文シークレットをログやファイルに置かない運用も必要です。

この 3 つだけでも、「とりあえず全部自動化」に倒さず、どこまで任せるかをユーザーに選ばせる思想が読み取れます。逆に言えば、選ばされている以上、使う側には設定を理解する材料が要ります。その材料置き場として、このガイドを作りました。

## 「いつの情報か」が分かるようにする

開発の速いツールのドキュメントは、鮮度の管理がすべてだと思っています。なので、運用ルールも最初に決めました。

- 全ページが対象バージョンと確認日をメタデータとして持つ
- 実機・ソースで確認済みのページは `verified`、製品更新後に差分確認が必要なものは `needs-review` として明示する
- アプリ更新後は、バージョン記録 → 画面の撮り直し → 索引との照合 → 既定値の再取得、の順で再調査する

非公式ガイドが公式ドキュメントに勝てる点があるとすれば、ここだと思っています。全項目の網羅と、「この記述はいつ・何を根拠に書かれたか」の明示です。
