Anthropic は、エージェント開発・コンテキスト設計・ツール設計についての知見を、エンジニアリングブログと Claude Code の公式ドキュメントで数多く公開しています。

この記事は、それらから設計判断に効く原則を20個抜き出し、短く整理したものです。各項目は「主張＋根拠」だけに絞り、末尾に出典をまとめました。一次情報を当たるときの索引として使ってください。出典は Anthropic のエンジニアリングブログ5本と、Claude Code / プロンプトの公式ドキュメント3本です。

:::message
各原則には「**担保**」＝この原則を実際に効かせるには、どこで守るのが正解かを1行で添えました。見方は次の3つです。

- **内在化済み**: Claude Code（モデル＋ハーネス）が既定で行う、または機構を持つ。利用者は知っておけばよく、改めて書く必要はない
- **正本に書く**: そのツール実装・評価・プロンプト・設定など、対象物のある場所に書く（常時ロードの CLAUDE.md ではなく現場側）
- **CLAUDE.md に書く**: 常時ロードの恒久ルールとして書くと効く
:::

## コンテキストエンジニアリング

### 1. コンテキストは有限の注意予算 ─ 足すほど精度が逓減する

LLM には「注意予算（attention budget）」があり、トークンを 1 つ足すたびに予算が削られて精度が落ちていく（context rot）。だから入れる情報は「多いほど良い」ではなく、最小十分まで絞り込む。入れる前に「このトークンは精度に寄与するか」を問う。ただし必要な文脈まで削ると、過少コンテキストという逆の失敗を招く。

- **担保**: 内在化済み（モデルの性質。原理として知っておけばよい）
- **出典**: Effective context engineering for AI agents

### 2. 文脈はジャストインタイムで引く ─ 識別子を保持し実行時にロードする

全データを事前ロードせず、ファイルパス・保存クエリ・リンクといった軽量な識別子だけを保持し、ツールで実行時に必要分だけロードする。人間がブックマークを必要になってから開くのと同じ発想だ。事前ロードは速く予測可能だが使わないデータまで予算を食い、ジャストインタイムは省コンテキストだが実行時のツールコストと遅延が乗る。

- **担保**: 内在化済み（Claude Code は必要時にファイルを読む。自作エージェントなら実装側）
- **出典**: Effective context engineering for AI agents

### 3. コード実行を挟むと中間データをモデルに通さず処理できる

MCP を直接のツール呼び出しではなくコード API として渡すと、エージェントは自らコードを書いて実行環境内でフィルタ・集約し、中間結果をモデルのコンテキストに通さずに済む。ツール定義のオンデマンドロードと合わせ、150,000 トークンを 2,000 トークン（98.7% 削減）にした例もある。1 万行の表を実行環境で絞れば、モデルは 1 万行ではなく 5 行を見る。中間データが大きいほど効く。

- **担保**: 正本に書く（MCP / ツールの実装側）
- **出典**: Code execution with MCP

### 4. サブエージェントの価値は並列化ではなくコンテキスト汚染防止

サブエージェントを使う第一の理由は速度（並列実行）ではなく、主会話を「再び参照しない探索ノイズ」で埋めないことだ。別のコンテキストウィンドウで作業し、要約だけを返すことで、有限のコンテキストを守る。並列化はその副次的な効果にすぎない。検索結果・ログ・大量のファイル内容で主会話が溢れそうな副作業に向く。

- **担保**: 内在化済み（サブエージェント機構あり。使う判断は運用側）
- **出典**: Claude Code best practices

### 5. 長期タスクはコンパクション・メモ帳・サブエージェントを使い分ける

コンテキスト上限を超える長期タスクの継続戦略は 3 つあり、タスク特性で選ぶ。往復の多い対話には**コンパクション**（履歴の要約圧縮）、マイルストーンが明確な反復開発には**メモ帳**（外部メモへの書き出し）、並列探索が効く調査・分析には**マルチエージェント**。排他ではなく、1 つの長期タスク内で局面ごとに組み合わせてよい。

- **担保**: 内在化済み（コンパクション等を内蔵。使い分けは運用判断）
- **出典**: Effective context engineering for AI agents

### 6. 長文プロンプトは文書を冒頭・クエリを末尾に置く

長い文書やデータはプロンプトの上部に、指示・クエリ・例はその下（末尾）に並べる。複数文書の複雑な入力では、クエリを末尾に置くだけでテスト上**最大 30%** の品質改善が見られた。末尾の指示を読む時点で、参照すべき文書がすでに文脈に展開されているため、注意が「何をすべきか」へ集中しやすくなる。

- **担保**: 正本に書く（その長文プロンプトの組み方）
- **出典**: Claude prompting best practices

## ツール設計（ACI）

### 7. ツール設計はプロンプトと同格の一次作業

エージェントの精度を律速するのは、プロンプト以上にツール定義とドキュメント ─ すなわち ACI（agent-computer interface）の品質である。Anthropic は人間向け UI と同じだけの工数を ACI に投じよと説き、SWE-bench エージェントでは「プロンプト全体より、ツールの最適化に多くの時間を費やした」という。相対パスでミスを繰り返したツールを絶対パス必須に変えただけで完璧に動いた例も示される。指針は「モデルの立場に立つ」。

- **担保**: 正本に書く（ツールの実装・定義）
- **出典**: Building Effective Agents

### 8. ツールは API の1対1ラッパーでなくタスクの自然な単位で作る

既存 API を 1 ツール＝1 エンドポイントで機械的に写すのではなく、頻繁に連鎖する複数操作を 1 ツールに束ね、タスクの自然な単位で設計する。`list_users`・`list_events`・`create_event` を個別に持たせるより、空き時間を見つけて予定を入れる `schedule_event` を 1 つ作る。薄いラッパーの寄せ集めは、エージェントに余計なオーケストレーションとラウンドトリップを押し付ける。

- **担保**: 正本に書く（ツールの実装）
- **出典**: Writing effective tools for AI agents

### 9. ツールは人間が使い分けを即答できる粒度まで絞る

機能を詰め込みすぎた、あるいは使い分けが曖昧なツール群は、エージェント失敗の典型である。「ある状況でどのツールを使うべきか人間のエンジニアが断定できないなら、AI エージェントにそれ以上は期待できない」。この「人間が即答できるか」を設計テストにして、重複・過剰なツールは削るか統合する。

- **担保**: 正本に書く（ツールの選定・実装）
- **出典**: Effective context engineering for AI agents

### 10. ツール応答の識別子は UUID より自然言語名にする

ツールが返す ID や参照は、不透明な英数字 UUID よりも、意味のある自然言語名（または 0 始まりの連番）にする。可読名に解決するとハルシネーションが減り、検索タスクの精度が大きく上がる。後続の呼び出しで ID を参照する際、人間可読な名前のほうがモデルが取り違えにくく、存在しない ID を捏造しにくいからだ。

- **担保**: 正本に書く（ツールの実装）
- **出典**: Writing effective tools for AI agents

### 11. ツールのエラー応答は次の一手を示す改善指示として返す

ツールがエラーを返すとき、生のエラーコードやトレースバックをそのまま返さず、エージェントが次に取れる具体的で実行可能な修正を伝える。エラー応答もプロンプトの一部として設計する対象だ。「パスは絶対パスで指定してください」のように、何をどう変えれば成功するかを明文で返す。デバッグ向けの詳細スタックトレースは、人間には有用でもエージェントの文脈を汚す。

- **担保**: 正本に書く（ツールの実装）
- **出典**: Writing effective tools for AI agents

### 12. ツールの評価は数十回の呼び出しを要する現実的複雑度で設計する

ツールの良し悪しは、単純すぎるサンドボックスではなく、現実のデータ・サービスに基づき数十回のツール呼び出しを要する複雑なタスクで測る。簡単すぎる評価は、ツールを改善しても劣化させても結果が動かず、変更の効果を見えなくする。評価タスクの現実性と複雑度が、そのまま測定の信頼性を決める。

- **担保**: 正本に書く（評価タスク・テスト）
- **出典**: Writing effective tools for AI agents

## エージェントとワークフローの選択

### 13. ワークフローとエージェントはサブタスクを事前特定できるかで選ぶ

LLM を組み込んだシステムは、処理経路を人間がコードで事前定義する「ワークフロー」と、LLM が自分のプロセスとツール使用を動的に決める「エージェント」に分かれる。必要なサブタスクを事前に予測できるならワークフロー、できないならエージェントを選ぶ。ワークフローは予測可能・低コスト・デバッグしやすく、エージェントは柔軟だがコスト・レイテンシ・不確実性が増す。

- **担保**: 正本に書く（アーキテクチャの設計判断）
- **出典**: Building Effective Agents

### 14. マルチエージェントは並列価値がコストを上回るタスクにだけ使う

マルチエージェントは単一エージェントの数倍〜十数倍のトークンを食う（自社データでエージェントは通常チャットの約 4 倍、マルチエージェントは約 15 倍）。「賢いから常に使う」のではなく、並列化で得られる価値がコスト増を上回るタスクにだけ割り当てる。独立した方向を同時に探索する breadth-first 調査は向き、依存が多くリアルタイム協調が要るコーディングは向かない。

- **担保**: CLAUDE.md に書く（起動前に効かせたい運用判断）
- **出典**: How we built our multi-agent research system

## プロンプト・指示の与え方

### 15. ルール列挙より多様で典型的な3〜5件の例で教える

起こりうるエッジケースのルールを延々と列挙するより、期待する挙動を表す多様で典型的な例を 3〜5 件厳選して示す方が効く。「例は千の言葉に値する絵」であり、エッジケースの羅列をプロンプトに詰め込むのは推奨されない。例は典型性（その挙動の代表になっているか）と多様性（守備範囲を散らせているか）で選ぶ。

- **担保**: CLAUDE.md に書く（恒久ルール／指示の書き方の作法）
- **出典**: Effective context engineering for AI agents

### 16. 指示には理由を添えるとルール外のケースも汎化する

禁止や指示に「なぜそうするか」を添えると、Claude が意図を理解し、明示していないケースにも正しく汎化する。「省略記号を使うな」より「TTS エンジンが読み上げるので省略記号を使うな（発音できないから）」の方が効く。裸のルールは書いた事例しかカバーしないが、理由つきは未記載の事例も拾う。ただし理由が誤っていれば誤った方向に汎化する。

- **担保**: CLAUDE.md に書く（恒久ルール／指示の書き方の作法）
- **出典**: Claude prompting best practices

## Claude Code の運用

### 17. CLAUDE.md は文脈であり強制ではない ─ 強制はフックで課す

CLAUDE.md の記述は Claude が文脈として扱うもので、強制設定ではない。判断次第で無視されうる。確実性が要る境界（削除・本番反映・秘密の出力など）は PreToolUse フックや権限で機械的にブロックし、緩い方針づけ（流儀・トーン・優先順位）は CLAUDE.md で足りる。

- **担保**: 正本に書く（フック・settings の権限）※「CLAUDE.md では担保できない」が要点
- **出典**: Claude Code memory

### 18. CLAUDE.md は肥大化すると従順度が落ちる ─ 追加より削除

CLAUDE.md は常時ロードされるため、長くなるほどコンテキストを食い、実際の命令への従順度がかえって下がる。各行に「これを消すと Claude が間違えるか」を問い、否なら削る。目標は **200 行未満**。運用の既定は「追加」ではなく「削除」で、新しい要件が来ても、まず削れる行から探す。

- **担保**: CLAUDE.md に書く（CLAUDE.md 自体の保守ルール）
- **出典**: Claude Code best practices

### 19. エージェントの完了条件は検証コマンドのパスで定義する

エージェントは「完了したように見える」時点で止まる。実行できる合否チェックがないと「looks done が唯一のシグナルになり、あなた自身が検証ループになる」。テストや Stop フックのような機械判定可能な pass/fail に完了を外部化すれば、自律ループが自走する。検証がないタスクには、人間が毎回成果を確かめる隠れた検証コストが残る。

- **担保**: 正本に書く（テスト・Stop フック）
- **出典**: Claude Code best practices

### 20. 同じ修正が2回続いたらコンテキストをリセットする

1 セッション内で同じ問題を 2 回を超えて訂正したら、文脈はすでに失敗したアプローチで汚染されている。`/clear` して、学んだこと（試して駄目だった方針・判明した制約）を取り込んだより具体的なプロンプトで再開する。良いプロンプトでのクリーンなセッションは、訂正を積み増した長いセッションをほぼ必ず上回る。

- **担保**: 内在化済み（人間の運用判断。仕組み化は不要）
- **出典**: Claude Code best practices
