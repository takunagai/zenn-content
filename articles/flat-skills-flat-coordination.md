汎用性のある粒度の小さいスキルを組み合わせて使い回せると嬉しい。「リサーチ → 要約 → レビュー」のように、小さなスキルをパイプラインで繋げられれば、スキルの再利用性が上がる。そこで最初に検討したのがオーケストレーター型だった。しかし、それは断念し、ユーザー仲介型を採用した。

**オーケストレーター型**: 司令塔となるスキルが複数のスキルを自動で順に呼び出す方式。LangChain や CrewAI などのエージェントフレームワークでは広く採用されている設計パターンだけど、Claude Code のスキルシステムには適合しなかった。

**ユーザー仲介型**: 各スキルが独立して動作し、完了時に「次にこのスキルを使いますか？」と提案する方式。オーケストレーター型との差は、ユーザーが「はい」と答える1ステップだけ。

## 前提: スキルシステムはフラット構造

Claude Code のスキルは `.claude/skills/{スキル名}/SKILL.md` に配置された Markdown ファイル。各スキルの `description` にキーワードを書いておくと、ユーザーの発話に最も合致するスキルを Claude が選んで実行する。

```yaml
---
name: contract-review
description: |
  契約書の要点を抽出し、リスク条項を指摘、チェックリストを生成する。
  「契約書をレビューして」「NDAのリスクを確認して」で発動。
---
```

**スキル同士に優先順位はない。** 全スキルがフラットに並び、同じ土俵で description のマッチングを競う。「このスキルを先に評価する」「このスキルが他を呼び出す」という仕組みはそもそも用意されていない。

公式の設計では、スキル間のデータ受け渡しやフロー制御は Claude 自身が担う。Claude がユーザーの発話と各スキルの description を照合し、最適なスキルを選んで実行する。つまり、**オーケストレーションの主体は Claude であり、スキルではない。**

ここにオーケストレーター型スキルを追加すると、Claude が description を参照する際にオーケストレーターと個別スキルが競合し、スキル選択を誤る可能性が高まる。

## オーケストレーター型が機能しない3つの理由

### 1. 発動競合

オーケストレーターの description をどう書くか。広くすれば単独で十分なリクエストまで奪い、狭くすれば連携が必要な場面で発動しない。スキルの数が増えるほど破綻する。

### 2. 非決定論的ルーティング

一般的なフレームワークのオーケストレーターはコードで実装されており、条件が一致すれば 100 回中 100 回同じ結果を返す。一方、スキルのオーケストレーターは Markdown の指示にすぎない。「contract-review → summarizer の順で実行して」と書いても、Claude がその通りに動く保証はない。

**コードの決定論的ルーティング**と**LLM 推論の非決定論的ルーティング** — 同じ設計名でも、載せる基盤が違えば機能しない可能性がある。

### 3. 既存 UX の破壊

例えば、契約書チェックスキルで「契約書をレビューして」→ contract-review が直接起動。シンプルで予測可能。オーケストレーター型はこの間にレイヤーを挟む。動いている UX を壊してまで導入するメリットは感じられなかった。

## ユーザー仲介型が機能する理由

contract-review 完了時の例:

> → 「高リスク条項について deep-research で調査しますか？」
> → ユーザーが「はい」→ 実行
> → ユーザーが「いいえ」→ 何も起きない

「はい」の1ステップと引き換えに、以下が手に入る。

- **発動競合ゼロ**: 各スキルの description は完全に独立
- **制御権はユーザーの手の中**: 不要な連携が勝手に走らない
- **既存 UX がそのまま**: 単独利用の体験を壊さない
- **スケーラビリティ**: 新スキル追加時に他スキルへの影響ゼロ

ただし制約もある。提案できるのはスキルの処理が完了した後だけなので、**連携は常に直列になる。** スキル A の途中で別のスキル B を呼び出し、その結果を A に戻して続行する、といった割り込み型の連携はできない。

用途としては「リサーチ → 要約 → レビュー」のような順序が明確なパイプラインに向いている。

## まとめ

オーケストレーター型は、決定論的ルーティングが可能な環境では正しい設計パターン。しかし、Claude Code のスキルシステムはフラット構造で、ルーティングは LLM の推論に委ねられているため、スキル連携には「ユーザー仲介型」の採用が妥当である。
