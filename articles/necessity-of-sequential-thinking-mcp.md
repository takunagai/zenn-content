開発環境をメンテナンスしていて、今まで入れていた [Sequential Thinking MCP](https://github.com/modelcontextprotocol/servers/tree/main/src/sequentialthinking) の必要性に疑問を感じたので整理してみました。

- 最近の Claude Code や Antigravity といった AIエージェントは、同じような処理をやってくれているのでは？
- 追加すると、むしろ品質が落ちそうで怖い

この記事では、「現在も Sequential Thinking MCP がを使った方がいいのか？」を整理します。  
今回出した結論としては、**採用しない方が合理的**です。

## Sequential Thinking は何か？

Sequential Thinking は、**思考プロセスをステップとして外に出し、扱いやすくするためのMCPサーバー**です。推論を“賢くする”というより「思考の扱い」を整えるツールです。

- 問題をステップに分解（Thoughtの積み上げ）
- 理解が進むにつれて、過去ステップを改訂（Revision）
- 代替案を分岐して探索（Branch）
- 必要ステップ数の動的調整、仮説の生成・検証 など

要するに「考えの進行を構造化し、記録・修正・分岐できる」という性格のツールです。

## それ、最近のエージェントはやってるんじゃ？
### Antigravity は、実務上ほぼ同じ体験を提供
- Artifacts（計画、タスクリスト、記録など）で進捗の検証・レビューができる
- 複数エージェントを並行稼働し、Managerビューで管理できる
- よって、Sequential Thinking を採用する理由は薄い

### Claude Code も、実務上ほぼ同じ体験を提供
- PlanモードでToDoを作成して進捗を管理できる
- 「計画して」と指示すれば、ステップごとの実装案を提案してくれる
- よって、Sequential Thinking を採用する理由は薄い

## 導入で逆に品質が下がったりしないか？

### リスク1：ステップ強制が探索を浅くする場合がある
モデルが内部でやっている圧縮・跳躍・統合を、外部の段階構造が過度に縛ると、
- 冗長化（“ちゃんと考えてる感”だけ増える）
- 寄り道探索の減少
- 統合的な判断の鈍化
が起きる可能性があります。

### リスク2：二重管理でノイズが増える
別レイヤーの思考構造を足すことで、「状態が二重化する」「ログが散らかる」「手戻りポイントが増える」といった余計な運用コストが発生します。

## 「プロトコル標準化」と「実装の完成度」は別物

Sequential Thinking の主張はざっくり言うと、以下のような方向性です。

- 実装（フレームワーク）ごとの“流儀”ではなく
- 思考ステップ（Thought/Revision/Branch）を
- 「共通の道具立て」として扱えるようにする

ただし開発者が欲しいのは、規格どうこうより勝てる実装です。

## 結論：いまは要らない（理解した上で使わないのが合理的）

次に当てはまるなら、現時点での導入メリットはないでしょう。

- Antigravity や Claude Code 等で、計画→実行→検証の型が既に定着している
- 思考ログの相互運用（他社/他実装への受け渡し）を必要としていない

Sequential Thinking は「新しい推論能力」ではなく「思考の外部構造を扱いやすくする」道具です。  
しかし、Antigravity のような統合実装や、Claude Code のような実務ベストプラクティスがある環境では、同等の体験が既に達成されていることが多いです。

その状態で追加導入すると、運用リスクや質の低下を招き、メリットよりデメリットの方が大きくなる可能性があります。したがって、現時点の多く開発者にとっては、「理解した上で、いまは採用しない」が最適解だと考えます。

> “Don’t fix what isn’t broken.”（壊れていないものを直すな）
