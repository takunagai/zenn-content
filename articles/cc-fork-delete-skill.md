## はじめに

Claude Code の `/branch`、使っていますか。

いまの会話を途中から枝分かれさせて、別のセッションとして続けられる機能です。元の会話はそのまま残るので、「ここまでの趣旨とずれる会話をしたい」「ここから少し違うことを試したい」というときに気軽に分岐できます。

ただ、気軽に分岐できるということは、**使い捨てのフォークが溜まっていく**ということでもあります。そして溜まったフォークを手で消すのは面倒だし危ない。そこで、フォークのセッション ID を渡すだけで安全に消すスキルを作りました。

この記事では、フォークの便利な使い方を紹介したうえで、その後始末をどう安全な仕組みにしたかを書きます。`/branch` を多用していて、溜まったフォークを安全に片づけたい Claude Code ユーザー向けの内容です。

## フォーク（/branch）とは ─ 気軽に分岐でき、使い捨てが溜まる

### 仕組み

`/branch` は Claude Code の組み込みコマンドです。チャット欄に打つと、その時点までの会話を引き継いだ**新しいセッション**が即座に作られます。新しいセッションには新しいセッション ID が振られ、元のセッションは手つかずで残ります。

Git のブランチをイメージすると分かりやすいです。ある地点から枝分かれして、本筋を汚さずに別の話を進められる。気に入らなければ、その枝ごと捨てればいい。

### 便利な場面

私がよく使うのは、次のような場面です。

1. **捨てる前提の危険な実験を隔離する** — 大きめのリファクタや、破壊的なコマンドの素振り。失敗しても元の文脈は無傷のまま。
2. **別アプローチを並行で比べる** — 「案 A」と「案 B」をそれぞれフォークで進めて、良かったほうを採用する。
3. **脱線質問を分ける** — 「そういえばこのライブラリの使い方は」のような寄り道を別セッションに逃がし、本筋の文脈を汚さない。（軽いものなら `/btw` で）
4. **長い議論の「ここから別案」** — 煮詰まってきた設計議論の途中から、別の方向を試す。
5. **フォークの中身を、あとから参照する** — フォークのセッション ID を指定すれば、その会話内容を元セッションや別プロジェクトから読み込めます。

特に 5 番目が、フォークを「使い捨て」以上のものにします。Claude Code のセッションは `~/.claude/projects/.../<セッションID>.jsonl` に残っていて、ファイルの場所はセッション ID から特定できる。だから分岐して得た結論だけを本筋に引き戻したり、捨てる前に成果をすくい上げたりできます。フォークは隔離された実験場であると同時に、結論を持ち帰る作業場にもなる。

持ち帰ったら、あとは捨てる。ここで後始末の問題が出てきます。

## 課題: 溜まったフォークを手で消すのは危ない

放置したフォークが増えるほど、`/resume` のピッカーは使い捨ての残骸で埋まり、目的のセッションを探すのが遅くなります。私はさらにセッションを別途索引化して管理しているので、その索引にも、実験のたびに切ったフォークの行が混じってくる。

では手で消せばいいかというと、これが気を使います。1 つのセッションは、複数の場所にファイルを残すからです。

- 会話本体: `~/.claude/projects/<encoded>/<セッションID>.jsonl`
- サブエージェントのログ: 同じ場所の `<セッションID>/` ディレクトリ（使った場合）
- セッションのメタ情報: `~/.claude/sessions/` 配下の JSON（`/rename` で付けた名前などが入る。ファイル名の規則は公式には公開されておらず、実データで確認したもの）
- （全文ログや索引を別途保存している場合）その出力先にも

加えて、こんな落とし穴があります。

- **いま自分が中にいるセッションは消せない**。稼働中の会話ファイルは開かれたままで、消しても終了時に書き直される。
- **短い ID で指定すると、別のセッションや元のセッションまで巻き込みかねない**。

「実験して、要らなくなったら捨てる」を気持ちよく回すには、この後始末こそ仕組みで安全にしておきたい。

## 解決: セッション ID を渡すだけの安全削除スキル

作ったのは `cc-fork-delete` という Claude Code のスキルです。使い方はシンプルで、

> 「フォーク `<セッションID>` を削除して」

と頼むと、次の流れで進みます。

1. スキルが **dry-run のプレビュー**を出す（対象セッション / 残すフォーク元 / 削除対象の一覧）
2. 内容を確認して、実行を承認する
3. 既定では**ゴミ箱（`~/.claude/.trash/`）へ退避**。完全削除ではないので、間違えても戻せる

設計の方針として、**実際の削除はすべて決定的なシェルスクリプトが行い**、Claude は「プレビューを出す → 確認をとる → 実行する」という進行役に徹します。消す・消さないの判断を、その場の AI の解釈に委ねないための切り分けです。

スキル本体（`delete-fork.sh` と `SKILL.md`）と、使い方をまとめたドキュメントは GitHub で公開しています。

https://github.com/takunagai/agent-assets/tree/main/skills/cc-fork-delete

https://github.com/takunagai/agent-assets/blob/main/docs/cc-fork-delete.md

## 安全設計: 「狙ったものだけ」を「取り返しがつく形」で消す

このスキルの中身は、ほとんどが安全ガードです。役割で 2 つに分けて説明します。

### 狙ったものだけを消す（誤爆防止）

**フォークかどうかを判定する。** フォークの会話ファイルの先頭行には、`forkedFrom` というフィールドがあります（公式ドキュメントには記載がなく、実データで確認したものです）。ここに「どのセッションから分岐したか」が入っている。これを見て、**`forkedFrom` を持たないセッション（＝大元）は削除を中止**します。「フォークだけ消す」を機械的に保証する仕組みです。

```bash
FORKED_FROM=$(head -1 "$JSONL" | python3 -c "import sys, json
ff = json.loads(sys.stdin.read()).get('forkedFrom')
print(ff.get('sessionId') if isinstance(ff, dict) else (ff or ''))")
```

この `forkedFrom`、実は実装中にハマったポイントでもあります（後述）。

**自分自身は消せないようにする。** 「稼働中のセッションは消せない」問題への対策です。Claude Code は現在のセッション ID を、公式に用意された環境変数 `CLAUDE_CODE_SESSION_ID` に持っています。これと削除対象を比べ、**一致したら中止**して「元のセッションに戻ってから実行して」と案内します。

```bash
if [ "$CLAUDE_CODE_SESSION_ID" = "$SID" ]; then
  echo "ERROR: 対象は現在実行中のセッションです。自分自身は削除できません。" >&2
  exit 3
fi
```

**厳密一致で対象を絞る。** 会話ファイルはフルのセッション ID で完全一致したものだけ。`sessions/*.json` は中身の `sessionId` をパースして厳密一致で判定します（ファイル名や文字列の部分一致では拾わない）。

**ログの誤爆を防ぐ。** 全文ログを残している場合、ここに罠があります。「ファイル内にそのセッション ID が含まれるログ」を素朴に拾うと、**たまたま会話本文でその ID に言及しただけの別セッションのログ**まで巻き込んでしまう。なので全文ログは、ヘッダ行に一致するものだけを対象にします。

```bash
grep -lE "^\*\*セッションID\*\*: $SID" "$LOG_DIR"/*.md
```

索引は、ファイルごと消すのではなく該当する行だけを削除し、ファイル自体はバックアップを取ってから編集します。

### 取り返しをつける（undo 系の安全弁）

**dry-run を既定にする。** `--apply` を付けない限り、スクリプトは何も消さず、削除対象を表示するだけです。必ず一度プレビューを挟む。

**ゴミ箱に退避する。** 既定の削除は `rm` ではなく、`~/.claude/.trash/forks/<sid>_<日時>/` への移動です。構造を保ったまま退避するので、後から戻せる。完全に消したいときだけ `--purge` を付けます。

## サンドボックスで検証する ─ forkedFrom の罠

破壊的なツールなので、作って終わりにはしませんでした。`HOME` を差し替えた**サンドボックス**を用意し、本物のデータには触れずに通します。

```bash
HOME=/tmp/sandbox CLAUDE_CODE_SESSION_ID=<別ID> ./delete-fork.sh <偽フォークID> --apply
```

このサンドボックスに「消すべきもの」と「残すべきもの」を両方仕込み、結果を突き合わせました。

- 消えるべき（フォークの会話ファイル / サブエージェント / メタ / ヘッダ一致の全文ログ / 索引の該当行）→ すべて退避された
- 残るべき（親セッションのメタ / 本文で ID に言及しただけの別ログ / 索引の親行）→ すべて保護された

ここで、先ほど触れた `forkedFrom` の罠です。最初はてっきり文字列だと思って実装していたのですが、実データは `{"sessionId": "...", "messageUuid": "..."}` という**オブジェクト**でした。そのまま表示すると、案内メッセージが「元へ戻る: `claude -r {'sessionId': ...}`」と崩れる。`sessionId` を取り出すよう直して解決しました。

公式ドキュメントに型が載っていない内部フィールドは、結局このように実データを開いて確かめるしかない。サンドボックスを先に組んでおいたおかげで、本番のセッションを壊さずにこの食い違いを拾えました。破壊的なツールほど、こうした「実データとのズレ」を安全な場所で先に潰しておく価値があります。

## まとめ

捨てる手段が面倒だったり怖かったりすると、結局フォークを作るのをためらってしまいます。後始末を仕組みにしてから、私は前より気軽に `/branch` を切るようになりました。失敗しても枝ごと捨てればいい——そう思えると、試す回数そのものが増えます。

コードとドキュメントは上記の GitHub リポジトリにあります。`forkedFrom` 判定・自分自身ガード・厳密一致・ログのヘッダ一致で「狙ったものだけ」に絞り、dry-run とゴミ箱退避で取り返しをつける、という構成です。出力先などを環境に合わせれば、そのまま使えます。
