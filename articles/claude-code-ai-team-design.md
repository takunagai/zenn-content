サブエージェントを増やすほど、AIチームは強くなる ── そのはずが、逆の現象が起きます。「この依頼はどのエージェントに？」が即答できなくなり、部品が増えるほどチームは動かしにくくなる。

強いチームを分けるのは機能の数ではなく、「誰が何を担うか」の地図と、それを仕組みで守る設計です。個別の機能は使えるがチームとして束ねる設計図がない ── そう感じている方に向け、5層モデル・設計の3原則・lint 自動化による齟齬ゼロの仕組みを示します。

## TL;DR

- Claude Code の機能は「役職 / スタッフ / スキル / コマンド / 協働」の5層で整理できる
- 設計の3原則: 役職は実装で軽く・地図と実装の齟齬を仕組みで守る・起動は絞る
- 組織図の正本（1ファイル）＋ PostToolUse hook lint で「増やしても破綻しない」状態を維持できる
- サブエージェントの第一の価値は「並列で速くなること」ではなく「コンテキストの隔離」

## なぜ「機能を覚える」だけでは足りないのか

Claude Code の機能は、抽象度も寿命もコンテキストの隔離度もバラバラです。サブエージェントは独立したコンテキストで動き、スキルは必要なときだけ読み込まれ、hook は設定ファイルに常駐します。これらを「便利な道具の寄せ集め」として個別に足していくと、道具は増えても、それらを束ねる組織が存在しないまま規模だけが膨らみます。

会社にたとえると、優秀なスタッフを採用し続けているのに、組織図も業務分掌もない状態です。一人ひとりは動けても、「この案件は誰の担当か」が決まっていない。これがサブエージェントを増やすほど管理できなくなる正体です。

必要なのは、機能を「役割の階層」に位置づけ直すことです。

## 1. メンタルモデル ── AIチームの5層

AIチームは、次の5つの層で考えると整理できます。混同しがちですが、それぞれ役割がはっきり異なります。

| 層        | 会社のメタファ    | Claude Code の機能         | 主な責務              |
| -------- | ---------- | ----------------------- | ----------------- |
| 1. 役職    | 部門・組織図     | プロジェクト/ユーザーのドキュメント（地図）  | どの文脈で誰を呼ぶかのルーティング |
| 2. スタッフ  | 専門社員       | サブエージェント                | 独立コンテキストでの専門作業    |
| 3. マニュアル | 業務手順書      | スキル                     | 再利用可能な手順・能力       |
| 4. 起動ボタン | 受付の呼び出し    | スラッシュコマンド               | 定型タスク・協働の起動       |
| 5. 協働・定例 | チーム編成・定例会議 | エージェントチーム / loop / hook | 並行作業・定期実行・自動チェック  |

設計が破綻する典型は、この層を取り違えることです。たとえば「役職」を独立したサブエージェントとして重く実装してしまうと、本来は軽いルーティングであるべきものが、毎回コンテキストを消費する重いペルソナ切り替えになります。逆に、本来コンテキストを隔離すべき長い専門作業を、メイン会話にすべて載せてしまうと、会話があっという間に肥大化します。

層を正しく当てはめるだけで、「何をサブエージェントにし、何をスキルにし、何をコマンドにするか」の判断がぶれなくなります。

## 2. 設計の3原則

5層を運用する上での土台になる原則が3つあります。すべての追加・変更はこの3原則に照らして判断します。

### 原則1 ── 役職は実装で軽く、機能で妥協しない

役職（部門）は「どの知識・スタッフ・判断軸を引き込むか」のスイッチに留めます。メイン会話に重い人格定義を読み込ませない。力は、独立コンテキストのサブエージェント・スキル・コマンドに持たせます。

「役職を持たせるとコンテキストを食う」という懸念は、役職を重いペルソナとして実装したときに限った話です。役職を軽い地図に留め、実作業を独立コンテキストへ逃がせば、組織化の利点（分担・専門性・並行処理）を、オーバーヘッドをほとんど払わずに得られます。

### 原則2 ── 地図と実装の齟齬ゼロを、仕組みで守る（保有は自由）

「誰が何を担うか」を書いた地図（組織図）と、実際のサブエージェント・コマンドの実体を、常に一致させます。重要なのは、この一致を **人の目視ではなく仕組みで保証する** ことです。

この一致が機械的に守られている限り、スタッフを何人抱えても運用者の認知は破綻しません。逆に言えば、「増やすと管理できなくなる」のは、地図と実装が手作業同期で、徐々にずれていくからです。同期を機械化すれば、保有数の上限はほぼ消えます。

### 原則3 ── 起動は規律する

保有が自由になっても、起動は別の話です。1つのタスクで多数のスタッフを同時に並行起動すれば、トークンとレイテンシは一気に膨らみます（Anthropic の自社データでは、単一実行と比べ 4〜15 倍のトークン消費になることがあります）。チーム化・並行起動は「並行で動かす価値がコストを上回るタスク」に限定します。

この3原則を一言でまとめると、「役職は軽く、地図は仕組みで守り、起動は絞る」です。

## 3. レイヤー別ベストプラクティス

### サブエージェント ── 独立コンテキストの専門スタッフ

サブエージェントの第一の価値は「並列で速くなること」ではなく **コンテキストの隔離** です。長大なリサーチ結果や草稿全体を読ませても、その消費はサブエージェント側のコンテキストに閉じ、メイン会話を圧迫しません。

設計で押さえるべきは3点です。

**tools は最小権限にする。** 書き換えが不要なレビュー系・批評系のエージェントには、`Read` / `Grep` / `Glob` だけを与えます。Write や Edit を持たせないことで、「判断」と「書き換え」を構造的に分離できます。

**model は判断の重さで選ぶ。** 重い判断（辛口の批評、難しいデバッグ）は上位モデル、定型・実装は中位モデル、軽作業は下位モデルに割り当てます。すべてを最上位にすると、コストが見合いません。

**description は「発火の一意性」を担保する。** description には次の3要素を必ず入れます。

```markdown
---
name: article-critic
description: >
  記事の読者視点での辛口レビューが必要なときに使う。
  5軸（独自性・読者解像度・具体性・読む価値・AI臭）で採点する。
  Do NOT: 構造の推敲（それは editor）、事実検証（それは fact-checker）には使わない。
tools: Read, Grep, Glob
model: opus
---
```

「いつ使うか」だけでなく **「いつ使わないか（Do NOT）」** を明記し、具体例をいくつか添える。たとえば `article-editor` と `article-critic` の両方に「推敲して」というトリガー語が入っていると、「この草稿を推敲して」という依頼でどちらを起動すべきか不定になります。`critic` が Write 権限を持っていれば、意図しない書き換えが走るリスクも生まれます。Do NOT を書いて責務を一意に絞ることで、こうした曖昧さを構造的に防げます。

### スキルとコマンド ── 委譲とパイプライン

スキルは「再利用する手順・能力」の置き場です。ここで重要な原則は **委譲** です。コマンドやサブエージェントから実作業を呼び出すとき、執筆やデプロイといった既存スキルの仕事を再実装してはいけません。コマンドは「順序と確認の制御」に徹し、実作業は既存スキルに渡します。

複数スタッフをつなぐ協働は、スラッシュコマンドで起動します。ここでのベストプラクティスは **ガイド付き順次実行** です。全自動で一気に走らせるのではなく、各フェーズの成果を提示してから次へ進む。

```markdown
---
description: 記事をリサーチから公開まで順に進める協働コマンド
argument-hint: [テーマ または ドラフトのパス]
---

各フェーズの成果を提示し、確認を取ってから次へ進む。
1. リサーチ（researcher サブエージェントに委譲）
2. 執筆（既存の執筆スキルに委譲。再実装しない）
3. 推敲（editor サブエージェント）
4. 批評（critic サブエージェント）
5. 事実検証（fact-checker サブエージェント）
```

創作やコードのように方向性に人の判断が要る作業ほど、フェーズ間に確認を挟む価値があります。全自動は、途中のズレを最後まで増幅させます。

### hook ── 齟齬を「仕組みで」守る

原則2「地図と実装の齟齬ゼロを仕組みで守る」を実現するのが hook です。`settings.json` の `PostToolUse` に、ファイル編集後に走るチェックを仕込みます。

ポイントは2つあります。1つは、対象ファイルを絞るガードを入れること。matcher はツール名またはそのパターン（正規表現）でしか絞れず、ファイルパスや引数の値では絞れないため、スクリプト側で「変更されたのが組織図ファイルのときだけ動く」よう制御します。もう1つは、問題を見つけたときの Claude へのフィードバック方法です。

```bash
#!/bin/bash
# 組織図と実体の齟齬を検出する lint（PostToolUse フック用）
INPUT=$(cat)   # フック入力は stdin から JSON で受け取る
changed=$(echo "$INPUT" | jq -r '.tool_input.file_path // empty')
case "$changed" in
  */roles/_index.md) ;; *) exit 0 ;;   # 地図ファイル以外は何もしない
esac

INDEX="$HOME/Documents/vault-main/roles/_index.md"
errors=""

# 死リンク: 地図に記したエージェントの実体が存在するか
while IFS= read -r path; do
  expanded="${path/#\~/$HOME}"
  [ -f "$expanded" ] || errors+="死リンク: $path\n"
done < <(grep -oE '`~/.claude/agents/[^`]+\.md`' "$INDEX" | tr -d '`')

# 孤立: 実体があるが地図に未登録のエージェントがないか
for f in "$HOME/.claude/agents/"*.md; do
  grep -q "$(basename "$f")" "$INDEX" || errors+="孤立: $(basename "$f")\n"
done

if [ -n "$errors" ]; then
  # PostToolUse フックでは exit code と stderr は無視される。
  # hookSpecificOutput.additionalContext を JSON stdout で返して Claude にフィードバックする。
  msg=$(printf "%s" "$errors" | tr '\n' ' ')
  printf '{"hookSpecificOutput":{"hookEventName":"PostToolUse","additionalContext":"地図と実装に齟齬があります: %s 修正してください。"}}' "$msg"
fi
exit 0
```

`hookSpecificOutput.additionalContext` に問題の内容を入れて stdout へ JSON で返すと、Claude がその内容を読んで修正を促せます。問題がなければ無音で `exit 0`。PostToolUse フックでは exit code と stderr は無視されるため、Claude へのフィードバックはこの JSON 形式で行います。地図を編集して実体との不整合を作り込んだ瞬間に、機械が気づかせてくれる ── それが原則2を実装で守る仕組みです。

:::message
既存の hook（フォーマッタなど）が `settings.json` に設定済みの場合は、同じ matcher の配列に追記します。設定後は JSON として壊れていないか必ず確認してください。1箇所の構文ミスで、登録済みの全 hook が動かなくなります。
:::

### エージェントチームと loop ── 起動の規律

複数エージェントの並行協働（エージェントチーム）や定期実行（loop）は強力ですが、原則3の「起動の規律」が最も問われる領域です。並行で動かすほどトークン消費は増えます。並行化は、独立した観点で同時に見る価値があるとき（多角的なレビューなど）に限定し、単純な逐次処理に安易に持ち込まないのが無難です。

定期実行も同様で、「人が毎回手で起こすのは面倒だが、放置すると溜まる」業務に絞ります。常時走り続けるループは作らない。これだけで、コストの暴走をかなり防げます。

## 4. 増やしても破綻させない仕組み

スタッフを増やしても破綻しないのは、次の3点がそろっているからです。

**組織図の正本を1つ持つ。** 「どの部門が・どのスタッフ／スキル／コマンドを・どんなトリガーで持つか」を、1つのファイルに集約します。これは人間が読む地図であると同時に、AI がトリガーから担当を引くルーティング表でもあります。情報源を二重に持たないことが肝心です。

実物の一部を示すとこうなります。

| スタッフ | 役割 | 主トリガー語 | 実体パス | 状態 |
|---|---|---|---|---|
| article-critic | 辛口批評・AI臭判定 | 辛口レビュー・AI臭 | `~/.claude/agents/article-critic.md` | 実在 |
| article-editor | 構成・冗長性の推敲 | 推敲して・構成見直して | `~/.claude/agents/article-editor.md` | 実在 |

実体パスを地図に明記することで、lint が「ファイルが本当に存在するか」を機械的に確認できます。

**齟齬を自動 lint で守る。** 前述の hook で、「地図にあるが実体がない（死リンク）」「実体があるが地図にない（孤立）」「トリガーの重複」を機械的に検出します。これが原則2を支えます。

**新メンバーの追加手順を標準化する。** 増やすたびに品質がぶれないよう、追加の手順を決めておきます。たとえば次のような流れです。

1. まず「作らない判断」をする ── 既存のスタッフ・スキル・組み込み機能で足りるなら作らない
2. tools 最小権限・model 適正・description 作法（使う/使わない/具体例）を満たす
3. 組織図の正本に、実体パス付きで登録する
4. lint を走らせ、死リンク・孤立・トリガー重複がないことを確認する

この3点があると、「保有は自由」が実際に成立します。地図が嘘をつかない限り、チームはいくら大きくしても見通せます。

## 5. 実装の勘所（Tips）

- **「作らない判断」を最初に置く。** 新しいサブエージェントを作る前に、既存スキルや組み込みコマンドで代替できないかを必ず確認します。重複は、地図と実装の齟齬を生む最大の原因です。
- **description は人間向けと AI 向けを兼ねる。** 曖昧な説明は、AI のルーティングを迷わせ、人の把握も妨げます。トリガー語を具体的に書き、他のスタッフと重ならないようにします。
- **実体パスを地図に書く。** 各スタッフの実体ファイルのパスを地図に明記しておくと、lint が機械的に存在確認できます。
- **設定ファイルを触ったら構文を即検証する。** `settings.json` のような JSON は、1文字の崩れで全体が無効になります。編集直後にパースが通るか確認する癖をつけます。
- **役職を増やすより、スタッフとスキルで吸収する。** 新しい業務領域が出ても、まずは既存部門にスタッフ／スキルを足して対応できないかを考えます。部門の新設は最後の手段です。

## まとめ

AIチームの設計でつまずく原因は、たいてい「部品を増やすこと」を目的にしてしまう点にあります。本当に設計すべきは、部品そのものではなく、それらを束ねる **組織** ── すなわち、誰が何を担うかの地図と、その地図を実装と一致させ続ける仕組みです。

- 機能は5層（役職・スタッフ・スキル・コマンド・協働）に位置づける
- 役職は軽く、地図は仕組みで守り、起動は絞る（3原則）
- 地図と実装の齟齬を自動 lint で守れば、保有数の上限はほぼ消える

地図が嘘をつかない仕組みを最初から仕込めば、スタッフを何人に増やしても運用者の認知は破綻しません。lint と正本化は、そのための最初の一手です。
