## TL;DR

- 「◯◯.app の△△の操作マニュアルを作って」という指示 1 行で、Claude Code が**アプリを実際に操作し、各ステップを撮影し、本文まで書き上げる**構成を作りました
- 実測では、5 ステップの手順に対して**操作 + 撮影が 2 分 30 秒、本文執筆まで含めて約 5 分**。連番スクショ 6 枚と完成マニュアル（Markdown）が日付フォルダに揃って出てきます
- 仕組みは 3 役分担です。**手 = 公式 computer-use**（Claude Code v2.1.85+ の組み込み MCP）、**カメラ = 自作の撮影ラッパー**（フル解像度・連番整理）、**筆 = Claude Code 本体**
- ハイブリッド構成ならではの罠も踏みました。**2 つの座標系を混同して、危うく隣のボタンを誤爆しかけた**話も書きます

対象読者は、クライアント納品や社内共有用の操作マニュアルを、スクショを 1 枚ずつ手で撮りながら作っている人。検証環境は macOS 15 + Claude Code v2.1.187（Max プラン）、題材アプリは Hermes Desktop v0.18.0 です。

## 仕組み ─ 手・カメラ・筆の 3 役分担

操作マニュアルの制作には、性質の違う 3 つの仕事が同居しています。アプリを正しい順序で操作する仕事、各ステップの画面を記録する仕事、そして読者向けの説明文を書く仕事。人間がやるときも、実際にはこの 3 役を 1 人で兼ねて、操作しては撮り、撮っては書き、を行き来しています。

Claude Code は次の分担で、この 3 役を同一セッション内で兼ねられます。

| 役割 | 使うもの | 選定理由 |
|---|---|---|
| 手（操作） | 公式 computer-use MCP | 縮小スクショ + zoom + サーバー側座標変換でクリック精度が高い。アプリ単位の承認 UI もあり操作の安全性が高い |
| カメラ（撮影） | 自作の撮影ラッパー `shot` | 公式のスクショは「モデルが見るための縮小画像」で成果物に不向き。自作ならフル解像度・ウィンドウ領域限定・連番で日付フォルダに整理できる |
| 筆（執筆） | Claude Code 本体 | 撮った画像を Read で見返しながら本文を書ける。操作した本人が書くので、画面と説明の食い違いが起きにくい |

「片方だけでよいのでは」と思うかもしれませんが、公式のスクショは承認外アプリが画面から消えてしまう（縮小もされる）ためマニュアルの図版にならず、自作ループだけで操作すると Retina 座標換算などの手数が増えます。操作と撮影で道具を分けるのが、実測してみて一番素直でした。公式 computer-use と自作ループの比較・使い分けの全体像は前作にまとめています。

https://zenn.dev/takna/articles/claude-code-mac-gui-diy-computer-use

## 前提環境

前作の環境構築（TCC 権限 2 つ + Swift ヘルパーのビルド、約 15 分）が済んでいる状態から始めます。追加で必要なのは、公式 computer-use の有効化（`/mcp` から Enable。Pro / Max プラン + v2.1.85 以降）だけです。

撮影ラッパー `shot` は数行の bash スクリプトで、撮った画像を `~/Downloads/GUIスクショ/<日時>/` に保存してパスを返します。30 分以内の連続撮影は同じフォルダにまとまるので、1 回のマニュアル作成 = 1 フォルダになります。

## 実録 ─ Cron ジョブ作成マニュアルを作らせた

題材は Hermes Desktop（Nous Research の AI エージェントアプリ）の Cron 機能。「決まった時刻にエージェントへプロンプトを自動実行させる」機能で、サンプルジョブを 1 つ作る手順をマニュアル化させました。

実際に投げた指示はこれだけです。

```
Hermes Desktop の Cron ジョブ(簡単なサンプル)を作成する手順で小さなマニュアル作成を試して。
補足: Cron の設定は左下の Cron から開くことができる。
```

補足 1 行を付けたのは、Cron 画面への入り口がステータスバーにあり、初見では見つけにくいと知っていたからです。この「人間が知っているコツを 1 行添える」だけで、探索の試行錯誤がまるごと消えます。

ここから Claude Code は次のように動きました。

1. **ウィンドウ座標を取得**して撮影領域を固定（以後の全スクショが同じ構図になる）
2. ホーム画面を撮影 → ステータスバーの「Cron」をクリック → ジョブ一覧画面を撮影
3. 一覧下の「+」をクリック → 新規作成ダイアログを撮影
4. 名前欄とプロンプト欄に**日本語で入力** → 入力済み状態を撮影
5. 「Cron を作成」をクリック → 一覧に追加されたことを確認し、作成したジョブの詳細画面を撮影

たとえば手順 2 で撮れたのがこの Cron 一覧画面です。左に登録済みジョブ、右に選択中ジョブの詳細が並びます（画面内の既存ジョブ名・履歴は無関係な内容のためぼかしています）。

![ステータスバーの「Cron」を開いたジョブ一覧画面。左が登録済みジョブ、右が選択中ジョブの詳細](/images/claude-code-app-manual-automation/step02_cron一覧を開く.png)

操作と撮影はここまでで **2 分 30 秒**。クリックはすべて一発命中、手戻りはゼロでした。

途中で効いたのが公式ツールの `computer_batch` です。「名前欄をクリック → 日本語入力 → プロンプト欄をクリック → 日本語入力 → 確認スクショ」という 5 アクションを 1 回のツール呼び出しにまとめられ、日本語もそのまま通りました。ここが 1 アクションずつだと、その分だけモデルの往復が増えていたはずです。

:::message
作成したサンプルジョブは本物の Cron として翌朝から毎日動き始めます。テストで作ったジョブは放置せず、一時停止か削除まで済ませるのを忘れずに。この「後始末」も含めてマニュアルの題材になりました（後述）。
:::

## 完成したマニュアル

撮り終えた 6 枚を Claude Code 自身が Read で見返しながら、本文を書き上げました。成果物フォルダの中身はこうなっています。

```
~/Downloads/GUIスクショ/2026-07-02_1537/
├── step01_ホーム画面.png
├── step02_cron一覧を開く.png
├── step03_新規作成ダイアログ.png
├── step04_内容を入力.png
├── step05_作成完了.png
├── step06_一時停止.png
└── manual.md
```

manual.md は「対象バージョン・所要時間 → 手順 1〜5（各手順に画像とポイント）→ 補足: ジョブの管理」という構成で、たとえば入力項目はこんな表に整理されていました。

> | 項目 | 内容 |
> |---|---|
> | 名前 | ジョブの表示名（省略可能） |
> | プロンプト | 実行ごとにエージェントが行う内容 |
> | 頻度 | 実行スケジュール。プルダウンから選択（既定: 毎日） |
> | 配信先 | 実行結果の届け先（既定: このデスクトップ） |

画像は本文から相対参照されているので、**フォルダごと渡せばそのまま配布物**になります。Markdown のままならリポジトリや社内 Wiki へ、Word 納品なら docx 変換、といった出口の変換だけが残る状態です。

実際に手順 4（名前・プロンプトを日本語入力した状態）で撮れた 1 枚がこれです。ダイアログ下部に「毎日 9:00 AM に」とスケジュールのプレビューが出ています（背後の一覧は無関係な内容のためぼかしています）。

![新規作成ダイアログに名前とプロンプトを日本語入力した状態。下部にスケジュールのプレビューが表示されている](/images/claude-code-app-manual-automation/step04_内容を入力.png)

## つまずき ─ 2 つの座標系を混同して空振り

本編 5 ステップは無傷でしたが、後始末フェーズでハイブリッド構成ならではの罠を踏みました。

サンプルジョブを一時停止しようと、ジョブ詳細画面の「一時停止」ボタンをクリックしたときのことです。クリックしたのに何も起きない。座標を確認すると、直前に `shot` で撮った**フル解像度画像のピクセル座標**を、そのまま公式 computer-use のクリックに渡していました。公式のクリック座標は「直近に公式ツールで撮った縮小スクショのピクセル値」で指定する仕様で、スケールがまるで違います。

結果は空振りで済みましたが、ずれた先にあったのは隣の「今すぐ実行」ボタン。押していればサンプルジョブが即実行されていました。実害のない題材だったから笑い話ですが、これが「削除」ボタンの隣なら話が違います。

対策はシンプルで、**「公式でクリックするなら、座標は公式のスクショから取る」を徹底する**こと。手（公式）とカメラ（自作）で画像の解像度が違う以上、座標の出どころを混ぜてはいけません。手元ではこのルールを方式判定用のスキルに書き込んで、構造的に再発しないようにしました。

なお、正しい座標でも 1 回目のクリックがホバー状態で止まり、2 回目で発火するという挙動も 1 度だけ観測しました。再現性は未確認ですが、「クリック後は必ずスクショで結果を確認し、変わっていなければもう一度」という基本動作でカバーできる範囲です。

## 運用の型 ─ そのまま使える指示テンプレ

今回の実測を経て、マニュアル作成の指示はこの形に落ち着きました。

```
<アプリ名> の「<作りたい手順>」の操作マニュアルを作って。
操作しながら各ステップを撮影して、本文まで書いて。
補足: <入り口や前提のコツを 1〜2 行>
```

あわせて、撮影まわりの規約を決めておくと成果物が安定します。

- **ウィンドウは 1 つのモニタに置いてテスト中は動かさない**（撮影領域を最初に 1 回固定するため。マルチディスプレイでもこれで問題なし）
- ファイル名は `stepNN_<操作名>.png` の連番。撮影ラッパー側で日付フォルダに自動整理
- 撮影前に、対象アプリ内に**機密（API キーが見える画面など）が写り込まないか一巡確認**しておく
- 破壊的な操作（削除・送信・課金）が手順に含まれる場合は、その操作の直前に確認を挟むよう指示しておく

## 応用と限界

**ウェブアプリでも同じ構図が組めます。** 部品が入れ替わるだけです。操作は chrome-devtools MCP（DOM 直接操作なので座標問題自体が消える）、撮影はページ内容だけなら同 MCP の `take_screenshot`、ブラウザの枠ごと見せたいなら `shot` でウィンドウ領域を撮る。ログイン済みの実ブラウザをそのまま使えるので、SaaS の管理画面マニュアルにも向きます。

**アップデート時の再撮影が一番おいしい応用です。** 手順が確定しているので、2 回目以降は探索が要りません。アプリの UI が変わるたびに「同じマニュアルを撮り直して、変わった箇所を報告して」と投げれば、画像の一括更新と差分検出が同時に済みます。

一方で、線を引いている限界もあります。

- **完全無人化はしない**。無人で許すのは検証済みスクリプトの再生と読み取り専用の撮影だけで、LLM がその場で座標を決める無人クリックは組みません
- 破壊的操作を含む手順は、その箇所だけ人間の確認を挟みます
- スクショは縮小されても文字が読める形で会話ログに永続化されます。機密が映る画面を扱うマニュアルでは、撮影範囲の設計を先にやるべきです

## まとめ

- 操作マニュアル作成は「手 = 公式 computer-use、カメラ = 自作撮影ラッパー、筆 = Claude Code 本体」の 3 役分担で一気通貫にできた。5 ステップの手順で、操作 + 撮影 2 分 30 秒、本文込み約 5 分
- 人間の仕事は「入り口のコツを 1 行添える」ことと、完成物の添削に変わる。スクショを撮って縮小して貼って、の反復作業は消える
- ハイブリッド構成の罠は座標系の混同。「公式でクリックするなら座標は公式のスクショから」を守れば防げる
- アップデートのたびに苦行だった「全スクショ撮り直し」が、指示 1 行の再実行になる。マニュアルは書くものから、**させて添削するもの**になりました
