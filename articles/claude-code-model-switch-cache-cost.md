以前の私は、一つのセッションの中でモデルとエフォートをこまめに切り替えていました。調べものや決まった処理など深い思考が要らないものは Sonnet に落とし、開発での実装の難所には Opus (使えた期間は Fable) に上げ、深く考えさせたいターンにはエフォートを xhigh に。タスクごとに最適な設定を選ぶ、賢い使い方をしている気になっていました。

これは誤りでした。切り替えるたびに、それまで積み上げたプロンプトキャッシュを捨て、会話履歴の再計算費用を払い直していたからです。

きっかけは、Claude Code でモデルを切り替えようとしたときに出るこの警告でした。

```text
Switch model?
 Your next response will be slower and use more tokens

 This conversation is cached for the current model. Switching to Fable 5 means
 the full history gets re-read on your next message.
```

「キャッシュがあるのに、どうして full history を読み直すのか」。調べて分かったことを、結論から書きます。

## 結論

- プロンプトキャッシュが省いているのは**通信ではなく計算**。切り替えは、その計算のやり直し ＝ 課金のやり直し
- **モデル切り替えはキャッシュを全部壊し、エフォート切り替えは会話履歴分のキャッシュを壊す**。どちらも「履歴の再計算」という隠れた支払いつき
- ただし再計算は一回きりの出費。**残りのセッションでずっと使うなら、切り替えて償却するのが正解**。高くつくのは、数ターンごとの上下動
- サブエージェント委譲も**安くする技法ではない**。固定費があり、小さいタスクでは回収できない

前提を 3 つ。料金は 2026-07 時点の公開単価（Sonnet 5 の $2/$10 は 2026-08-31 までの導入価格、以降 $3/$15）。Claude Code の既定キャッシュ TTL は 5 分。筆者が課金額を実測した記事ではなく、公式ドキュメントと API 仕様からの整理です。

## 仕組み ─ キャッシュされているのはテキストではなく計算結果

キャッシュの実体は、送ったテキストではなく、**そのモデルの重みで計算した KV（key/value）テンソル** 。会話履歴を forward pass にかけたときの中間状態です。

たとえるなら、会話履歴（毎回送るテキスト）は分厚い資料、モデルはそれを読む担当者、KV は**読み終えた担当者の頭の中にできた理解**です。キャッシュが取っておくのは、資料のコピーではなく、担当者の理解のほう。

モデルを切り替えると、担当者が別人になります。資料は同じでも、前任者の頭の中の理解は引き継げません。新しい担当者が最初から読み直すしかない ─ 警告文の "the full history gets re-read" は、文字どおりの意味でした。公式マイグレーションガイドも、モデル文字列を変えると既存キャッシュは無効化され、新モデルで書き直しになると明記しています。つまり警告文は「キャッシュがあるのに使わない」ではなく、**「新しいモデルにとっては、これは初回リクエストと同じ」**と言っているわけです。

### 毎回全文を送っているのに、なぜ安くなるのか

Messages API はステートレスで、クライアントは毎リクエストで会話履歴を丸ごと送ります。100 ターン目でも、1 ターン目からの全履歴がワイヤーを流れる。それなのにキャッシュヒット分の課金は 1/10 になる ─ 一見矛盾しています。

種明かしは、課金が見ているのが転送バイト数ではなく**サーバーが消費した計算量**であることです。資料を手渡すこと（通信）は一瞬で、コストの本体はそれを読み込んで理解を組み立てること（**prefill** ─ 全入力トークンへの forward pass）のほう。サーバーは受け取ったプロンプトの前半が前回と一致していれば、そこまでの理解が頭に残っているので、続きだけ読めば済みます。省いているのは通信ではなく計算です。

キャッシュがサーバー側にあることは、API レスポンスの `usage` から確認できます（値は説明用の例）。

```json
"usage": {
  "input_tokens": 300,
  "cache_creation_input_tokens": 1500,
  "cache_read_input_tokens": 48000,
  "output_tokens": 900
}
```

`cache_read_input_tokens` は「サーバーが自分のキャッシュから拾えた分」の申告です。キャッシュがクライアント側にあるなら、サーバーがこの数字を報告できるはずがありません。

:::details 傍証（TTL / モデル別キャッシュ / cache_control）
- **TTL がある**（5 分 / 1 時間）。クライアントが持っているだけのテキストに expire の概念は要りません。サーバーが GPU メモリやストレージを解放するから TTL があります
- **モデルごとに別キャッシュになる**。クライアントがテキストを持っているだけなら、モデル依存になる理由がありません
- **`cache_control` はブレークポイントの指示にすぎない**。クライアントが送るのは「ここまでをキャッシュしてくれ」というマーカーだけで、キャッシュの実体は送っていません
:::

### 何を変えると、どこまで壊れるか

キャッシュは `tools`（ツール定義）→ `system`（システムプロンプト）→ `messages`（会話履歴）の階層で積まれ、**変更した層とそれ以降が無効化されます**（手前の層は残る）。

| 変更するもの | tools | system | messages | 焼ける範囲 |
|---|---|---|---|---|
| モデル | ✘ | ✘ | ✘ | 全部（KV はモデル固有） |
| ツール定義 | ✘ | ✘ | ✘ | 全部 |
| エフォート / thinking（有効・無効・budget） | ✓ | ✓ | ✘ | messages だけ |
| tool_choice / 画像の増減 | ✓ | ✓ | ✘ | messages だけ |

エフォート（`low`〜`max`）は、新しいモデルでは thinking の深さを制御するパラメータです。公式ドキュメントは「thinking の設定変更（有効・無効・budget）は会話履歴のキャッシュを無効化する」と明記しています。

注意したいのは、**「messages 層だけ」でも十分に痛い**ことです。system と tools は大きくても固定サイズ、messages は会話が進むほど育つ大物。長いセッションでは「履歴の読み直し」という一番重い部分をそのまま食らいます。

:::message
エフォート値の変更がキャッシュを壊すかは、厳密には推論を含みます。公式が明記しているのは「thinking パラメータの変更」であり、エフォート単体を名指しした記述は見つけられませんでした。thinking を動かす以上、messages キャッシュは壊れるとみるのが自然、という整理です。
:::

### 料金への効き方

キャッシュの単価は、基本入力単価に対して cache read が **0.1 倍**、cache write が 1.25 倍（5 分 TTL）または 2 倍（1 時間 TTL）。出力トークンはキャッシュと無関係に常に全額です。Claude Code はブレークポイントを自動で前進させて履歴をインクリメンタルにキャッシュするので、長いセッションの入力は「0.1 倍の巨大な前半 + 1 倍の小さな差分」になります。

ただし**安くなるだけで、短くはなりません**。履歴のトークン数も、コンテキストウィンドウの消費も、auto-compact の発火タイミングも変わらない。「キャッシュが効くからツール出力を大量に積んでも平気」は成り立たず、流し込んだ raw dump は以後ずっとウィンドウを占有し続けます。

レート制限にも効きます。API の ITPM（input tokens per minute）は cache read トークンをカウントしません（公式に「Cache-aware ITPM」として明記。Haiku 3.5 のみ例外）。

## 切り替えの支払いは 2 種類。一回きりの読み直しと、その後ずっとの継続費用

切り替えた瞬間に払うのは、履歴の読み直し（cache write）です。モデル切り替えなら全層、エフォート切り替えなら messages 層。これは**一回きり**で、済めば以降はまたキャッシュが効きます。

見落としやすいのは継続費用のほうです。100k の履歴を抱えたセッションの 1 ターンあたり入力コストは、Sonnet 5 継続で 100k × $0.20 = **$0.02**、Opus 4.8 継続で 100k × $0.50 = **$0.05**。差は $0.03/ターン。一方、一回の読み直しは 100k × $10（Opus の 1 時間 TTL cache write）= **$1.00** なので、**入力差だけでも約 33 ターンで読み直し分を追い越します**。出力単価も $10 対 $25 で開くため、実際はもっと早い。

エフォートの場合、継続費用の中身が違います。モデルが変わらないので単価は上がらず、代わりに**毎ターンの思考トークン生成量が増える**。モデルは単価、エフォートは思考量 ─ レイヤーは違いますが、どちらも「一回の読み直し」の先にかかり続ける費用です。

ここから運用の軸が出ます。**残りのセッション全体で上位モデル（または高いエフォート）が要るなら、迷わず切り替える**。読み直しは後のターンで償却されます。高くつくのは「数ターンだけ上げてすぐ戻す」を繰り返すこと。… 冒頭の私の使い方そのものです😅

## 委譲は「安くする技法」ではない

「あと少しだけ上位モデルを使いたい」場合の代替が、サブエージェントへの委譲です。親はそのままに、サブエージェントを `model` 指定で起動してそのタスクだけ任せる。ただし委譲のコストを「引き継ぎプロンプトの分だけ」と数えるのは間違いで、実際の支払いはこうなります。

| 項目 | 誰の単価で払うか |
|---|---|
| 引き継ぎプロンプトの執筆 | 親の**出力**トークン |
| サブエージェントの system prompt + ツール定義 | **上位モデルの入力**（固定費） |
| 再発見（親が既に読んだファイルの読み直し） | 上位モデルの入力 |
| サブエージェント自身のターンループ | 上位モデル（探索するほど積む） |
| サブエージェントの出力 | 上位モデルの**出力**（最も高い区分） |
| 親が結果を取り込む | 親の入力 + cache write（以後ずっとコンテキストを占有） |

特に効くのが **system prompt とツール定義の固定費**です。新規エージェントは毎回ゼロから立ち上がるので、タスクが小さいほど固定費が割高になる。Anthropic 自身のデータでも、単一エージェントはチャット対話の約 4 倍、マルチエージェントは約 15 倍のトークンを消費します。委譲は安くする技法ではなく、**トークンを積んででも文脈と単価を分離する技法**です。

:::message
2026-07 時点の Claude Code では、サブエージェント起動時に**モデルは指定できてもエフォートは指定できません**（Task ツールに effort 引数がない）。委譲先のエフォートまで変えたいなら Workflow の `agent()` が必要です。バージョン依存の挙動です。
:::

### 損益分岐

委譲が親の切り替えより安くなる条件は、不等式ひとつに落ちます。

**引き継ぎプロンプト P + 再発見コスト D ＜ 会話履歴 H**

引き継ぎは親の出力、履歴の読み直しは上位モデルへの cache write。1 時間 TTL なら Sonnet の output 単価 $10 と Opus の cache write 単価 $10 が一致し、比較はトークン数だけの勝負になります（この一致は Sonnet 5 の導入価格前提で 2026-09 以降は崩れ、既定の 5 分 TTL なら分岐は `P + D < 0.625 × H` にずれますが、P と D が小さければ結論は同じです）。

効いてくるのは左辺の **D** です。サブエージェントは親が読んだファイルやツール結果を持たず、必要なら上位モデルの単価で読み直します。タスクが会話文脈に依存するほど D が膨らみ、優位が消える。しかも Claude Code では**文脈を丸ごと継承しつつモデルだけ上げることができません**。コンテキストを完全継承する `fork` は model 指定を無視するため、上位モデルを使うには手書きの引き継ぎしかなく、**引き継ぎは必ず lossy になります**。

### 判断軸

**そのまま親で続ける** ─ タスクが小さく、委譲の固定費も切り替えの読み直しも回収できない。現行モデルで解ける見込みがある

**親を切り替える** ─ 残りのセッション全体で上位モデルが要る（読み直しが償却される）。タスクが会話文脈全体を要求する。まだ序盤で履歴が小さい

**サブエージェントに投げる** ─ 難所が「重いが境界は明確」で短いプロンプトで書ける。直後に安いモデルへ戻る。必要な情報を会話履歴でなくファイルから再導出できる

エフォートも同じ物差しです。残りずっと深い思考が要るなら上げっぱなしで償却、ピンポイントならその数ターンだけ。ただし上下動のたびに messages キャッシュが焼き直しになるので、こまめに動かさず「ここぞ」でまとめて上げる。

そして単価より優先すべき判断がひとつ。**lossy な引き継ぎで答えを間違えたら、やり直しのコストが節約分を軽く上回ります。** 委譲は「この文脈は失っても大丈夫か」で先に切るのが安全です。

## 運用チェックリスト

- **モデルもエフォートも、切り替えるなら序盤に寄せる**。履歴が数万トークンに育ってからが一番高い
- **数ターンごとの上下動をやめる**。上げるなら償却できる長さで、まとめて
- **用途でモデルを分けるなら `/clear` してから**。捨てる履歴を高い単価で書き直さない
- **元の設定に戻すなら TTL（既定 5 分）内に**。キャッシュが生きていれば再課金されない
- **枠を守る最大のレバーはモデル選択そのもの**。キャッシュ最適化より、どのモデルで走らせるかが効く

## まとめ

- キャッシュの実体は KV テンソル。モデルの重みに紐づき、モデルをまたいで共有できない
- 毎回全文は送られている。省かれているのは通信ではなく **prefill という計算**
- 安くはなるが短くはならない。コンテキストウィンドウの消費は変わらない
- モデル切り替えは全層、エフォート切り替えは messages 層を焼く。どちらも「一回きりの読み直し」と「その後ずっとの継続費用」の 2 段構え
- 残りが長いなら切り替えて償却、あと少しなら動かさない。委譲は安くする技法ではなく、文脈と単価を分離する技法

課金の直感が外れるのは、こちらがバイト数で数えているのに、課金は計算量で数えているからです。この一点さえ握っておけば、モデルもエフォートも委譲も同じ物差しで測れます。冒頭の警告文が言っていたのも、結局「新しい担当者に、資料を最初から読み直させますよ」でした。
